## [프로젝트] 공동거주시설 통합 관리 플랫폼: Home Talk One-Pass

**1. 주차 시스템 기획 의도 및 확장성**
- 범용적 설계: 대단지 아파트뿐만 아니라 경비 인력이 적은 소규모 빌라, 오피스텔 등 다양한 공동 거주시설에 즉시 도입 가능한 통합 관리 솔루션 지향.
- 유연한 확장성: 현재는 경비원의 수동 입력을 지원하지만, 향후 LPR(번호판 인식) 장비 연동 시 데이터 소스만 교체하면 즉시 자동화 시스템으로 전환 가능한 하드웨어 독립적 모듈 설계.

---

**2. 사용자 역할 및 권한**
- 관리자: 입주자 차량 등록 증빙 서류 검토(승인/반려), 세대별 입주자 승인, 전체 주차 기록 조회
- 입주자: 내 차량 등록(1:N, 증빙 서류 첨부 필수), 반려 시 사유 확인 및 재등록, 방문 차량 사전 예약, 주차 중인 방문 차량에 티켓 등록(1시간/1일권).
- 현장 관리자(경비): 통합 대시보드(퀵 서치 및 예약 목록)를 통한 실시간 입/출차 기록 및 신원 식별.


**차량 구분**
- 입주자 차량: 관리자 승인 완료 차량
- 방문 차량: 사전 예약 차량
- 수동 입차 차량 : 예약 없이 현장에서 경비원이 직접 등록한 차량 (초기 상태: 수동입차 - 미등록 방문) 이후 입주민이 세대 매칭 가능

**3. 주차 시스템 상세 동작 프로세스**
[1단계: 입차 전 사전 예약 및 차량 등록]
- 내용: 입주민이 앱에서 '내 차량'을 등록하거나 '방문 차량'의 번호, 목적, 예정 시간을 예약함.
- 시스템: 로그인 정보를 기반으로 방문 세대 정보를 자동 매칭하여 저장.

[2단계: 입차 처리 (데이터 기반 신원 식별)]
- 내용:
    차량 진입 시 경비원이 차량 번호를 입력(퀵서치 활용)하면,
    시스템은 전체 등록 데이터와 대조하여 차량을 식별한다.
- 신원 식별 결과:
    시스템은 차량 번호를 기준으로 [입주자 차량 / 사전 예약 차량 / 미등록 차량]으로 자동 분류한다.
    - 입주자 차량:
        "결과: [OOO호] 입주자 차량입니다." → 즉시 입차 승인
    - 사전 예약 차량:
        "결과: [OOO호] [방문 목적] 방문 차량입니다." → 확인 후 입차 승인
    - 미등록 차량:
        "결과: 등록되지 않은 차량입니다. 목적 확인 후 입차하세요." → 수동 입차 처리
        · 미등록 차량 처리 로직:
            미등록 차량으로 입차한 경우, 초기에는 세대 정보 없이 주차 기록이 생성되며
            'UNREGISTERED' 상태로 관리된다.
            이후 입주민 또는 경비원이 해당 차량을 특정 세대에 매칭할 수 있으며,
            매칭 완료 시 해당 주차 기록은 정상 정산 및 알림 대상에 포함된다.
- 시스템 로직:
    입차 승인 시점의 시간을 입차 시간으로 DB에 자동 기록하고, 차량 상태를 '주차중(PARKED)'으로 자동 변경한다.
- 실시간 알림 발송:
    · 발송 시점: 입차 승인 처리 완료 직후
    · 알림 대상: 해당 차량과 연결된 세대 회원 전체
    · 알림 채널: 웹 & 앱 푸시 알림

[3단계: 출차 전 티켓 등록 및 정산]
- 내용:
    입주민이 앱에서 현재 주차 중인 차량을 확인하고, 보유한 티켓(1시간권 / 1일권)을 선택하여 적용한다.
- 시스템 로직:
    시스템은 입차 시간을 기준으로 현재까지의 주차 시간을 자동 계산하며, 적용된 티켓 시간을 반영하여 정산을 수행한다.
- 티켓 처리 방식:
    티켓은 매월 1일 세대별로 자동 발급되며, 사용 시 보유 수량에서 차감되고 주차 기록 ID와 매칭되어 사용 이력이 관리된다.
- 사용자 제어:
    입주민은 출차 전까지 티켓 적용을 취소하거나 수정할 수 있다.

[4단계: 출차 검증 및 완료]
- 내용:
    출구에서 경비원이 차량 번호를 조회하면,
    시스템은 (현재 시간 - 입차 시간)과 적용된 티켓 시간을 자동 계산하여 정산 결과를 도출한다.
- 최종 처리:
    - 정상:
        티켓 적용 범위 내일 경우 즉시 [출차 완료] 처리되며, 해당 세대에 출차 알림이 발송된다.
    - 초과:
        '시간 초과(OVERSTAY)' 상태로 표시되며, 해당 세대에 알림을 발송하여 티켓 추가 적용 또는 현장 조치를 유도한 후 출차 처리된다.

**4.요구사항 분석** 
주차 관리 시스템 - 요구사항 명세서 (간략)
1) 업무 개요
공동거주시설에서 **회원(입주민)**이 차량을 등록하고, 방문 차량을 예약 및 정산하는 과정을 관리한다.

[차량 구분 기준]
- 입주자 차량: 관리자 승인 완료된 차량
- 방문 차량: 사전 예약 또는 미등록 상태로 입차한 차량

2) 상세 요구 사항
R1. 회원은 이름, 연락처, 동/호수(고유) 정보를 가진다.
R2. 차량은 전체 차량번호(고유), 모델명 정보를 가진다.
R3. 한 명의 회원이 여러 대의 차량을 등록할 수 있다. (1:N)
R4. 하나의 세대는 여러 건의 방문 예약을 생성할 수 있다. (1:N)
R5. 입/출차 시 입차시간, 출차시간, 상태, 사용 티켓 정보를 기록한다.
R6. 주차 상태는 "입차예정", "주차중", "출차완료", "시간초과" 중 하나이다.
R7. 시스템은 매달 1일 세대별로 정해진 수량의 티켓을 자동 발급한다.
R8. 티켓 발급 정책은 단지별 설정에 따라 발급 수량 및 주기가 달라질 수 있다.
R9. 시스템은 티켓 기반 정산 외에도 결제 기반 정산 방식으로 확장 가능해야 한다.
R10. 주차 요금이 발생한 경우, 사용자는 티켓 적용 또는 결제 방식 중 하나를 선택할 수 있다.
R11. 결제 기능은 외부 결제 시스템과 연동 가능하도록 설계되어야 한다.
R12. 초과 요금 발생 시, 현장 결제 또는 사전 결제를 통해 정산할 수 있어야 한다.

3) 제약조건
C1. 차량 번호는 시스템 내 중복 불가 -> UNIQUE 제약조건
C2. 주차 시간 계산은 분(Minute) 단위로 수행 -> CHECK 제약조건
C3. 회원 삭제 시 해당 세대의 차량 등록 정보는 삭제 -> ON DELETE CASCADE
C4. 입차 시간은 반드시 출차 시간보다 빨라야 한다. -> CHECK (입차 < 출차)
C5. (추가 제안) 티켓 잔여 수량은 0보다 작을 수 없다. -> CHECK (잔여량 >= 0)
C6. 티켓 발급 수량 및 정책은 단지별 설정 값을 따른다.
C7. 정산 방식은 티켓 또는 결제 방식 중 하나 이상을 지원해야 한다.

5. 기능 요구사항 (Requirements)
1) [차량 및 방문 관리]
R1. 입주자는 본인 소유 차량을 시스템에 등록할 수 있어야 한다.
R2. 등록된 차량은 관리자 승인 후 활성화되어 입출차 및 알림 서비스 대상이 된다.
R2-1. [신규] 차량 등록 증빙 및 승인 프로세스
- 서류 첨부: 입주자는 차량 등록 시 자동차 등록증 또는 보험 가입 증명서 등 증빙 서류를 사진/파일 형태로 반드시 업로드해야 한다.
- 승인 대기: 서류가 업로드된 차량은 즉시 활성화되지 않고 '승인 대기(Pending)' 상태로 저장된다.
- 관리자 검토: 관리자는 관리자 페이지에서 업로드된 서류를 확인 후 [승인] 또는 [반려] 처리를 수행한다.
- 승인 시: 해당 차량의 상태가 **'입주자 차량'**으로 활성화되며, 이때부터 자유로운 입출차 및 알림 서비스가 제공된다.
- 반려 시: 관리자는 반드시 **반려 사유(예: 서류 식별 불가, 정보 불일치 등)**를 작성해야 하며, 해당 입주민에게 즉시 반려 알림이 발송되어야 한다.
R3. 한 명의 입주자는 여러 대의 차량을 등록할 수 있어야 한다. (1:N)
R4. 입주자는 방문 차량의 전체 차량번호, 방문 목적, 방문 예정 시간을 사전 예약할 수 있어야 한다.
R5. 시스템은 차량 예약 시 현재 로그인된 입주자의 세대 정보(동/호수)를 데이터베이스에 자동으로 연결하여 저장해야 한다.
R6. 경비원 페이지는 당일 입차 예정 리스트를 기본 제공한다.
	경비원이 차량 번호 뒷자리(4자리) 입력 시, 해당 번호를 포함한 예약 차량 및 입주자 차량을 실시간으로 필터링하여 리스트업한다.
R7. 리스트에 중복된 번호가 있을 경우, 경비원은 화면에 표시된 전체 번호와 방문 세대 정보를 대조하여 정확한 차량을 선택할 수 있어야 한다.
R8. 입주자는 앱 내에서 본인 세대의 차량 및 방문 차량의 과거 주차 이력을 월별로 필터링하여 조회할 수 있어야 한다.


2) [입/출차 및 상태 관리]
R9. 경비원은 차량 입차 시 차량 번호를 검색하여 등록 차량 여부를 확인할 수 있어야 한다. (단, 입주자 차량은 관리자 승인이 완료된 차량만 검색 결과에 노출된다.)
	- 검색 방식: 경비원이 차량 번호 뒷자리 4자리만 입력하면, 시스템은 현재 '입차 예정(예약)' 및 '입주자 등록' DB에서 해당 숫자를 포함한 모든 차량을 실시간으로 리스트업한다.
	- 신원 확정: 리스트에 여러 대가 뜰 경우, 경비원은 차량의 전체 번호(앞자리 포함)를 대조하여 정확한 차량을 선택하고 [입차 승인]을 진행한다.
	- 데이터 연결: 선택된 차량은 DB에 저장된 전체 차량 번호와 세대 정보(동/호수)가 즉시 연결되어 안내 메시지를 출력하고 알림을 발송한다.
R10. 시스템은 차량 번호 조회 시 다음 정보를 표시해야 한다.
	-입주자 차량 여부 : 결과: [OOO동 OOO호] 입주자 차량입니다. (승인 버튼 활성화)
	-방문 예약 여부 : 결과: [102동 405호] 예약 차량(가족 방문)입니다. (승인 버튼 활성화)
	-방문 세대 정보 : 결과: 등록되지 않은 차량입니다. 방문 목적 확인 후 수동 입차를 진행하세요. (경고 색상 표시)
R11. 경비원이 입차를 승인하면 시스템은 해당 시점의 시간을 입차 시간으로 자동 기록해야 한다.
R12. 차량이 입차하면 시스템은 주차 상태를 ‘주차중’으로 변경해야 한다.
R13. 시스템은 차량의 주차 상태를 다음 상태 값으로 관리해야 한다. [입차예정, 주차중, 출차완료, 시간초과]
R14. 시스템은 차량의 입/출차 및 시간 초과 이벤트 발생 시 해당 세대 입주민에게 실시간 알림을 전송해야 한다.
	R14-1 (입차 알림): 입주자 및 예약 차량이 단지 내 진입하여 경비원이 입차를 승인하는 순간, "차량(번호)이 입차하였습니다"라는 알림을 전송한다.
	R14-2 (출차 알림): 차량이 정산을 마치고 출차 완료 상태로 변경될 때, "차량(번호)이 출차하였습니다"라는 알림을 전송한다.
	R14-3 (초과 알림): 방문 차량이 예약 시간 또는 티켓 허용 시간을 초과하여 주차 중일 경우, 출차 전 입주민에게 "주차 시간이 초과되었습니다. 티켓을 확인하세요"라는 조치 요청 알림을 전송한다.


3) [티켓 및 정산 관리]
R15. 입주자는 현재 주차 중인 방문 차량에 대해 주차 티켓을 등록할 수 있어야 한다.
R16. 주차 티켓은 ‘1시간권’과 ‘1일권’ 두 종류로 구성되며, 입주민이 티켓을 적용할 때마다 세대별 잔여 수량에서 즉시 차감한다.
R17. 시스템은 단지별 설정된 티켓 발급 정책에 따라 세대별 티켓을 자동 발급 및 초기화해야 한다.
R18. 출차 시 시스템은 다음 산식에 따라 주차 시간을 자동으로 계산하여 표시해야 한다.
	- 전체 주차 시간: (현재 시간 - 입차 시간)을 '분(Minute)' 단위로 환산.
	- 적용 티켓 시간: 입주민이 등록한 티켓의 총 합산 시간(분).
	- 미적용(초과) 시간: (전체 주차 시간 - 적용 티켓 시간). 결과가 0보다 작으면 0분으로 표시.
R19. 모든 시간 계산은 '분' 단위로 수행하되, 화면 출력 시에는 사용자가 직관적으로 알 수 있도록 'OO시간 OO분' 형태로 변환하여 표시해야 한다.
R20. 시스템은 (전체 주차 시간)이 (티켓 사용 시간)을 초과할 경우 ‘출차 제한’ 알림을 표시하고, 해당 세대에 알림을 전송한다.

4) [관리자]
R21. 시스템은 차량별 주차 이력을 조회할 수 있어야 한다.

5) [입주자 주차 이력 및 월간 관리]
R24. 입주자는 앱을 통해 본인 세대와 연결된 차량(입주자 등록 차량 및 방문 예약 차량)의 전체 주차 기록을 실시간으로 조회할 수 있어야 한다.
R25. 주차 이력 리스트에는 다음의 데이터가 반드시 포함되어야 한다.
	-차량번호: 전체 번호 표시
	-등록구분: '등록 차량' 또는 '방문 차량'으로 구분 표시
	-시간정보: 입차 및 출차 시간 (출차 전일 경우 '주차 중' 상태 표시)
	-정산내역: 방문 차량의 경우, 해당 기록에 사용된 티켓 정보(1시간/1일권) 표시
R26. 입주자는 조회 화면에서 '연도 및 월(Year-Month)'을 선택하여 해당 달에 발생한 주차 이력만 모아볼 수 있어야 한다.
R27. 보안을 위해 입주자는 본인이 소속된 세대의 데이터만 조회 가능하며, 타 세대의 주차 기록에는 접근할 수 없어야 한다.


[미등록 차량 사후 매칭 및 정산]
R28. 예약되지 않은 차량이 진입할 경우, 경비원은 차량 번호를 입력하여 확인 후 ‘수동 입차 차량’ 상태로 입차 승인할 수 있어야 한다.
시스템은 입차 시간과 차량 번호를 기록한다.
수동 입차 차량은 입주민이 세대 매칭을 수행할 수 있어야 한다.

R29. 매칭 후 티켓 적용 : 세대 매칭이 완료된 시점부터 입주민은 해당 차량에 대해 보유한 주차 티켓(1시간/1일권)을 적용할 수 있으며, 출차 시 자동 정산 대상에 포함된다.


R30. 수동 입차 차량이 특정 세대에 매칭되면 다른 세대에서는 조회되지 않는다.

[결제]
R31. 시스템은 티켓 기반 정산 외에도 결제 기반 정산 방식으로 확장 가능해야 한다.
R32. 주차 요금이 발생한 경우, 사용자는 티켓 적용 또는 결제 방식을 선택할 수 있어야 한다.
R33. 결제 기능은 외부 결제 시스템과 연동 가능하도록 설계되어야 한다.
R34. 초과 요금 발생 시, 현장 결제 또는 사전 결제를 통해 정산할 수 있어야 한다.
R35. 정산 로직은 티켓 및 결제 방식을 분리하여, 동일한 주차 기록에 대해 선택적으로 적용할 수 있도록 설계되어야 한다.
